Include statements

Rules related to #include statements.

Name Description Why is this important?

list-symbols

Requires #include statements to be suffixed with a comma-separated list of all the symbols used from the included header or file.

                
#include <stdbool.h>
                
                
#include <stdbool.h> // bool

You can indicate that you're using every symbol of a specific pattern by applying the star-character:

                
// Using all symbols starting with 'mylib_'
#include <mylib.h> // mylib_*

Helps in determining unused headers that could be removed and encourages use of smaller, more well-defined headers.

Fewer inclusions reduce complexity, so if an inclusion does not use any symbols, it should be removed.

If a symbol is used, but in a different source file, the inclusion should be moved to that file.

symbol-listed-not-used

Requires that any symbol listed as used by an #include statement should also be used in the same source file.

See list-symbols

Helps determining symbols that are no longer used, potentially leading to being able to remove the inclusion completely, improving maintainability.

symbol-used-not-listed

Requires that any used symbol be listed as used by an #include statement.

Not implemented as a check

Helps determining which symbols should be listed as used, which in turn improves maintainability.

Miscellaneous

Rules related to code in general.

Name Description Why is this important?

line-too-long

Requires lines to not exceed 80 characters.

Any line of code should fit on the screen it is being viewed on under any scenario; whether single file or side-by-side. Lines that are too long can be difficult to visually comprehend, and wrapping or scrolling makes it harder to read.

Lines shorter than 80 characters fits on most viewers and will improve readability significantly.

file-too-long

Requires files to not exceed 600 lines.

Files that are very long can be difficult to comprehend and may indicate that the complexity of the code is too high.

Considerations should be made whether code can be refactored or split up into smaller units.
